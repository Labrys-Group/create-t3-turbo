We have adopt an explicit naming convention for Next.js server actions so call sites immediately signal that they cross the client/server boundary rather than behaving like ordinary local functions. A simple rule such as prefixing actions with `server`, `run`, or `submit` and using verb-first names like `serverCreateOrder`, `submitLeadAction`, or `runCheckoutAction` makes the networked side effect obvious in both imports and usage. This improves readability, reduces mistaken assumptions about latency, error handling, and data freshness, and makes reviews easier because engineers can instantly see when code is invoking server-only logic instead of client state updates. 
The goal is not just consistency, but making the boundary legible so the code communicates cost, execution context, and risk without requiring someone to inspect the implementation.
